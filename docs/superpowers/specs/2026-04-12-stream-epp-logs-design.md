# Stream EPP Logs — Design Spec

**Date:** 2026-04-12  
**Status:** Draft

## Problem

EPP (Endpoint Picker Proxy) logs are extremely verbose — hundreds of lines per second of NDJSON. CRI-O's per-container log rotation limit (~10–50 MB) is exhausted in roughly one minute of EPP output. `kubectl logs` on a running pod only surfaces the current (unrotated) log file, so the `collect-results` task — which runs at end-of-workload — captures only the final ~1 minute of EPP activity. All earlier logs are permanently lost.

## Goal

Capture the full EPP log for every benchmark workload and store it on `data-pvc` in time-bucketed files for post-run analysis. The EPP pod must continue writing to stdout (live visibility in the OpenShift UI is a hard requirement — no in-pod file redirection).

## Non-Goals

- Real-time log search or querying during the run.
- Log collection for vLLM decode pods (already captured fully; they are less verbose).
- Changes to the EPP deployment or container configuration.

## Execution Model Context

The experiment pipeline (generated by `make_experiment_pipeline` in `pipeline/lib/tekton.py`) runs all (phase, workload) groups **sequentially** in a single combined PipelineRun. Each group is prefixed: `baseline-fm3-burst-*`, `baseline-overload-*`, `treatment-fm3-burst-*`, etc. Within a group, EPP is deployed fresh (`deploy-gaie`), the workload runs, logs are collected (`collect-results`), and then EPP is deleted (the converted `finally` tasks). The next group then deploys a fresh EPP.

This means at any point in time, at most one EPP deployment is active. However, **multiple workloads share the same phase name** (e.g., both `baseline-fm3-burst` and `baseline-overload` write to `${runName}/baseline/`). The sentinel mechanism must use a **per-workload path** to avoid the second workload's streaming task exiting immediately on the stale sentinel written by the first.

## Design

### Approach: Parallel streaming Tekton task

A new task `stream-epp-logs` starts immediately after `deploy-gaie` and runs in parallel with the workload tasks. It follows EPP stdout via `kubectl logs --follow --timestamps=true` and splits the stream into 5-minute time-bucketed files on `data-pvc`. The task exits when `collect-results` writes a per-workload sentinel file signalling that this workload is complete.

This approach requires no cluster-level changes, no modifications to the EPP container, and integrates cleanly with Tekton's DAG execution model.

### New task: `stream-epp-logs`

**File:** `tektonc-data-collection/tekton/tasks/stream-epp-logs.yaml`

**Parameters:**

| Name | Type | Description |
|---|---|---|
| `namespace` | string | Kubernetes namespace |
| `modelLabel` | string | e.g. `sim2real-<experimentId>` |
| `resultsDir` | string | Per-workload results path on data-pvc, e.g. `<runName>/baseline/<workloadName>`. Same value as `collect-results` receives. |
| `windowMinutes` | string | Log rotation window in minutes (default: `"5"`, must be integer 1–60) |

**Workspace:** `data` (shared data-pvc)

**Derived paths (computed in script):**

```sh
RESULTS_DIR="/workspace/data/$(params.resultsDir)"
LOG_DIR="${RESULTS_DIR}/epp_logs"          # inside the workload directory
SENTINEL="${RESULTS_DIR}/epp_stream_done"  # per-workload → no collision between workloads
```

**Steps:**

1. **wait-and-stream**

   ```
   Phase 0 — Validation:
     WINDOW="$(params.windowMinutes)"
     If WINDOW is not an integer in [1, 60]: log error, exit 0 (non-fatal).

   Phase 1 — Pod discovery:
     RELEASE="${MODEL_LABEL}-gaie"
     EPP_DEPLOY="$(echo -n "${RELEASE}" | cut -c1-40)-epp"
     (Matches the truncation logic in deploy-gaie and collect-results.)
     Poll every 3 s (up to 60 s / 20 attempts) until:
       kubectl get pods -n $NAMESPACE --no-headers \
         -o custom-columns=NAME:.metadata.name | grep "^${EPP_DEPLOY}"
       returns at least one pod name.
     If no pod found after timeout: log warning, exit 0 (non-fatal).

   Phase 2 — Streaming:
     mkdir -p "${LOG_DIR}"
     PIDS=""
     For each EPP pod found:
       kubectl logs --follow --timestamps=true "${pod}" -n "${NAMESPACE}" \
         | awk -v outdir="${LOG_DIR}" -v pod="${pod}" -v win="${WINDOW}" '
           {
             # --timestamps=true prepends: 2026-04-11T23:28:39.123456789Z
             # $1 = timestamp prefix; $2..$NF = original JSON log line.
             # Extract hour and minute from the T-separated time component.
             split($1, dt, "T")
             split(dt[2], tm, ":")
             hour = tm[1]
             min  = int(tm[2])
             bucket = int(min / win) * win
             fname = sprintf("%s/%s_%s%02d.log", outdir, pod, hour, bucket)
             # Print only the original log line (strip kubectl timestamp prefix).
             out = ""
             for (i = 2; i <= NF; i++) out = out (i > 2 ? " " : "") $i
             print out >> fname
             fflush(fname)
           }
         ' &
       Record $! — the awk PID (rightmost process in the pipe). In bash,
       `A | B &` sets $! to B's PID. Killing awk breaks the pipe; kubectl
       receives SIGPIPE on its next write and exits cleanly.
       PIDS="${PIDS} $!"

   Phase 3 — Sentinel poll:
     Every 5 s:
       If "${SENTINEL}" exists: break
       If all tracked PIDs have exited (pod died/deleted): break
     On exit:
       kill ${PIDS} 2>/dev/null || true   # kills awk → SIGPIPE terminates kubectl
       wait ${PIDS} 2>/dev/null || true   # waits for awk to flush remaining lines
       Log "EPP log streaming complete for ${RESULTS_DIR}"
       Exit 0
   ```

   **Note on BusyBox awk:** The `alpine/kubectl:1.34.1` image uses BusyBox awk. It supports
   `split`, `int`, `sprintf`, `print >>`, and `fflush()` but does NOT support `strftime`,
   `systime()`, or `mktime()`. All timestamp parsing must use string manipulation only.
   The field splitting on the first whitespace (`$1` = timestamp prefix) is compatible.

**Image:** `alpine/kubectl:1.34.1` (already used in the pipeline).

### Log file layout

EPP log files live inside the workload results directory, matching the existing layout where per-workload decode logs and trace files already reside:

```
<runName>/<phase>/<workloadName>/
  epp_logs/
    <pod-name>_HH{bucket:02d}.log
  <pod-name>_tokenizer-uds.log      ← existing (non-epp containers, unchanged)
  <decode-pod>_vllm.log             ← existing decode logs
  trace_header.yaml
  trace_data.csv
```

Example (5-minute windows, UTC, `adaptive6` baseline `fm3_burst`):

```
workload_fm3_burst/
  epp_logs/
    sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_2325.log   # 23:25–23:29
    sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_2330.log   # 23:30–23:34
    sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_2335.log   # 23:35–23:39
```

Each workload group deploys a fresh EPP pod with a new random suffix, so `epp_logs/` files are always distinct across workloads. If multiple EPP replicas are running, each pod produces its own files in the same `epp_logs/` subdirectory.

### Modified task: `collect-results`

**Changes:**

1. Remove the EPP pod log-collection block (lines 45–71 in current script). The streaming task owns EPP logs.
2. At the end of the `collect-logs` step, write the per-workload sentinel:
   ```sh
   touch "/workspace/data/$(params.resultsDir)/epp_stream_done"
   ```
   This signals the concurrent `stream-epp-logs` task for this specific workload to exit.
3. No new parameters needed. `resultsDir` already exists and has the right per-workload scope.

The vLLM decode log collection block is unchanged.

### Modified pipeline template: `pipeline.yaml.j2`

Add task `stream-epp-logs` after `deploy-gaie`, running in parallel with the workload:

```yaml
- name: stream-epp-logs
  runAfter: ["deploy-gaie"]
  taskRef:
    name: stream-epp-logs
  workspaces:
    - name: data
      workspace: data-storage
  params:
    - name: namespace
      value: "$(params.namespace)"
    - name: modelLabel
      value: "sim2real-$(params.experimentId)"
    - name: resultsDir
      value: "$(params.runName)/{{ phase }}/$(params.workloadName)"
```

No changes to the `collect-results` task entry in the template (it already passes `resultsDir`).

### Execution flow and Tekton DAG safety

```
deploy-gaie
  ├── stream-epp-logs (follows EPP, writing to phase/epp_logs/) ──────────┐
  └── (other setup tasks)                                                   │
        └── run-workload                                                    │
              └── collect-results → touches resultsDir/epp_stream_done ────┘
                                                                            ↓
                                                          stream-epp-logs exits
                                                                            ↓
                                        (delete-gaie, delete-model, ... run)
                                                                            ↓
                                              Next workload group starts
```

For a two-workload phase (`fm3-burst`, then `overload`):

```
baseline-fm3-burst-deploy-gaie
  ├── baseline-fm3-burst-stream-epp-logs   sentinel: baseline/fm3-burst/epp_stream_done
  └── ... → baseline-fm3-burst-collect-results  writes ^ sentinel

baseline-fm3-burst-delete-gaie  (EPP deleted)

baseline-overload-deploy-gaie   (fresh EPP)
  ├── baseline-overload-stream-epp-logs    sentinel: baseline/overload/epp_stream_done
  └── ... → baseline-overload-collect-results   writes ^ sentinel
```

Sentinel names are distinct per workload, so the second workload's streaming task cannot
spuriously exit on the first workload's sentinel.

Tekton's `finally` block (in the individual phase pipeline) is converted to regular tasks
by `make_experiment_pipeline`, so EPP deletion is part of the sequential chain and cannot
race ahead of `stream-epp-logs`.

### Edge cases

| Scenario | Behaviour |
|---|---|
| EPP pod not found after 60 s | Log warning, exit 0 (non-fatal; `epp_logs/` may be empty) |
| EPP pod crashes mid-run | `kubectl logs --follow` exits naturally; that pod's stream ends; sentinel poll continues for the remaining duration |
| Sentinel never written (pipeline failure) | Streaming task runs until the Tekton task-level timeout configured in the pipeline |
| Multiple EPP replicas | All pods are streamed concurrently; each produces its own files in the shared `epp_logs/` directory |
| `kubectl logs` exits with API server error | Background process dies; sentinel poll detects all processes gone and exits 0 |
| `windowMinutes` is 0 or non-integer | Script validates the param at startup; logs an error and exits 0 with no streaming |
| `epp_logs/` directory cannot be created (PVC issue) | `mkdir -p` fails; script logs error and exits 0 (non-fatal) |
| Two workloads produce the same UTC hour bucket | Not an issue: each workload has its own `epp_logs/` directory inside its workload results dir, so files from different workloads never share a directory. |

### Impact on `deploy.py` / results collection

`_extract_phase_from_pvc` copies the entire `${runName}/${phase}/` tree from the PVC. The per-workload `epp_logs/` directories are already within that tree and are collected automatically with no changes to `deploy.py`.

After collection, the local layout matches the existing structure with `epp_logs/` added inside each workload directory:

```
run_dir/deploy_baseline_log/
  workload_fm3_burst/
    epp_logs/
      sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_2325.log
      sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_2330.log
    sim2real-adaptive6-gaie-epp-6dc8d58d99-r46pm_tokenizer-uds.log
    sim2real-adaptive6-decode-*_vllm.log
    trace_header.yaml
    trace_data.csv
    epp_stream_done          ← sentinel (small empty file, can be ignored)
  workload_fm5_short_output/
    epp_logs/
      sim2real-adaptive6-gaie-epp-6dc8d58d99-v55ml_2340.log
      ...
```

`epp_logs/` files sort lexicographically into chronological order.
